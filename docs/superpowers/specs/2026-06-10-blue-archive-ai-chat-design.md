# Blue Archive AI Chat — 精简二次元化设计文档

**日期**: 2026-06-10  
**项目**: Open WebUI 精简 + 二次元化  
**主题**: 蔚蓝档案 (Blue Archive) 风格  
**吉祥物**: 阿罗纳 (Arona) — 亮色模式 / 普拉纳 (Plana) — 暗色模式

---

## 1. 项目目标

将功能繁多的 Open WebUI 大幅精简为最小可用聊天应用，同时全面二次元化为 **蔚蓝档案 (Blue Archive)** 风格。

### 核心原则
- **YAGNI**: 只保留"能跑起来"的最小功能集
- **主题一致性**: 所有视觉、文案、交互都围绕 Blue Archive 世界观
- **明暗双模式**: 阿罗纳（亮色）/ 普拉纳（暗色）一键切换
- **HermesAgent 原生支持**: 深度绑定 HermesAgent，原生展示工具调用和思考过程

## 4. 明暗主题切换机制

### 实现方式

1. **Tailwind `dark:` 前缀 + CSS 变量**: 在 `:root` 定义亮色变量，在 `[data-theme="dark"]` 定义暗色变量
2. **根属性切换**: 在 `src/app.html` 或布局根组件注入 `data-theme` 属性
3. **切换按钮**: 顶部导航栏右侧，太阳/月亮图标，带平滑过渡动画
4. **持久化**: 主题状态保存到 `localStorage`
5. **默认行为**: 跟随系统 `prefers-color-scheme`，用户可手动覆盖

### 切换动画

- 切换时阿罗纳/普拉纳立绘有 0.5s 交叉淡入淡出
- 页面背景色渐变过渡 0.3s

## 7. 数据流

### 7.1 聊天数据流

```
用户输入
    ↓
前端 Chat 组件 → 显示用户气泡
    ↓
POST /api/v1/chats/completions
    ↓
后端验证 → 转发到配置的 LLM (Ollama/OpenAI API)
    ↓
SSE 流式返回
    ↓
前端逐字显示 AI 气泡（带阿罗纳/普拉纳立绘）
    ↓
流结束，气泡完成
```

### 7.2 文件上传流

```
用户选择文件
    ↓
POST /api/v1/files/
    ↓
后端保存文件，返回 file_id
    ↓
前端将 file_id 附带在下次聊天请求中
    ↓
LLM 根据文件内容回复
```

## 8. HermesAgent 深度集成

> **核心需求**: 项目必须深度绑定 HermesAgent，能够显示其 **工具调用过程** 和 **思考过程**。

### 8.1 HermesAgent 简介

[HermesAgent](https://hermesagent.agency/) 是 Nous Research 开发的开源 AI Agent 框架，核心特征：
- **ReAct 循环**: Observation → Reasoning → Action → 迭代直至完成
- **XML 工具调用**: 使用 `<tool_call>`、`<tool_response>` 标签格式
- **显式思考**: 使用 `<think>`、`<reasoning>` 标签输出思考过程
- **兼容 OpenAI API**: 通过标准接口接入，流式返回 SSE

### 8.2 消息格式解析

HermesAgent 的响应流中可能包含以下 XML 标签块：

```xml
<!-- 思考过程 -->
<think>
用户问的是天气，我需要调用 get_weather 工具。
首先确认城市名称...
</think>

<!-- 工具调用 -->
<tool_call>
<name>get_weather</name>
<arguments>{"location": "上海", "unit": "celsius"}</arguments>
</tool_call>

<!-- 工具返回（由后端/中间层注入） -->
<tool_response>
<name>get_weather</name>
<output>{"temperature": 26, "condition": "晴朗"}</output>
</tool_response>

<!-- 最终答案 -->
上海今天天气晴朗，气温 26°C~ 🌞
```

### 8.3 前端渲染设计

#### 8.3.1 思考过程区域 (`<think>` / `<reasoning>`)

```
┌──┬────────────────────────────────────────┐
│阿│ ┌─ 💭 思考过程 [展开 ▼] ──────────────┐│
│罗│ │ 用户问的是天气，我需要调用...        ││
│纳│ │ 首先确认城市名称...                  ││
│  │ └────────────────────────────────────┘│
│  │                                         │
│  │ 上海今天天气晴朗，气温 26°C~ 🌞        │
└──┴────────────────────────────────────────┘
```

- **默认状态**: 折叠，只显示 "💭 思考过程" 标题行
- **展开后**: 显示灰色/半透明背景的思考内容区域
- **样式**: 圆角 8px，背景色比主气泡更暗/更浅（区分层级）
- **角色化**: 阿罗纳显示 "💭 阿罗纳正在思考..." / 普拉纳显示 "💭 普拉纳分析中..."

#### 8.3.2 工具调用卡片 (`<tool_call>`)

```
┌─ 🔧 工具调用: get_weather ──────────────┐
│ 参数:                                    │
│ {                                        │
│   "location": "上海",                    │
│   "unit": "celsius"                      │
│ }                                        │
│                              [执行中 ⏳] │
└─────────────────────────────────────────┘
```

- **图标**: 🔧 扳手图标
- **标题**: "工具调用: {工具名}"
- **内容**: JSON 参数，带语法高亮
- **状态**: 
  - 执行中: 蓝色边框 + ⏳ 旋转动画
  - 成功: 绿色边框 + ✅
  - 失败: 红色边框 + ❌
- **可折叠**: 点击标题可展开/折叠参数详情

#### 8.3.3 工具返回卡片 (`<tool_response>`)

```
┌─ ✅ 工具返回: get_weather ──────────────┐
│ 结果:                                    │
│ {                                        │
│   "temperature": 26,                     │
│   "condition": "晴朗"                    │
│ }                                        │
└─────────────────────────────────────────┘
```

- **图标**: ✅ 或 📤
- **标题**: "工具返回: {工具名}"
- **内容**: JSON 结果，带语法高亮
- **默认**: 折叠状态，只显示标题行

### 8.4 流式解析状态机

前端 SSE 解析器需要维护以下状态：

```
States: TEXT → THINK_START → THINK_CONTENT → THINK_END
              → TOOL_CALL_START → TOOL_CALL_CONTENT → TOOL_CALL_END
              → TOOL_RESPONSE_START → TOOL_RESPONSE_CONTENT → TOOL_RESPONSE_END
```

**解析规则**:
1. 遇到 `<think>` 或 `<reasoning>` → 切换至思考模式，内容进入思考区域
2. 遇到 `</think>` 或 `</reasoning>` → 结束思考，切回正常模式
3. 遇到 `<tool_call>` → 创建新的工具调用卡片，等待完整 XML
4. 遇到 `<tool_response>` → 创建工具返回卡片，与上一个 tool_call 配对
5. 其余内容 → 正常 Markdown 渲染

## 11. 技术决策

### 10.1 为什么不移除 SvelteKit？

原项目的构建系统、路由、状态管理都基于 SvelteKit。移除它意味着重写整个前端架构，风险高且不符合"能跑起来"的目标。保留骨架并删除非核心目录是最务实的方案。

### 10.2 为什么不保留 RAG？

用户明确要求"不需要向量化"。RAG 是整个项目中最复杂的子系统之一（涉及向量数据库、文档解析、嵌入模型）。移除它可以大幅减少后端复杂度和依赖。

### 10.3 为什么不使用 Live2D？

用户明确要求"不需要 live2d"。静态立绘 PNG 更简单、性能更好、无需额外库。

### 10.4 HermesAgent 集成为什么是纯前端改造？

HermesAgent 兼容 OpenAI API 标准，通过 SSE 流式返回内容。后端只需做请求转发，无需理解 XML 标签语义。所有 `<think>`、`<tool_call>` 等标签的解析和渲染都在前端完成，这样：
- 后端改动最小（符合"精简"目标）
- 前端可以灵活控制展示样式
- 如果未来切换到其他 Agent 框架，前端渲染层可以复用

## 13. 风险与缓解

| 风险 | 缓解措施 |
|------|---------|
| 删除过多导致编译失败 | 分批次删除，每批删除后验证编译 |
| 依赖清理后还有残留 import | 使用 `grep` 扫描被删除模块的引用 |

| 精简后后端数据库模型不兼容 | 保留核心 models，删除无关表 |
| HermesAgent XML 格式变化 | 解析器设计为容错模式，未知标签按原文显示 |
| 工具调用卡片与流式渲染冲突 | 状态机管理渲染阶段，确保卡片不闪烁/跳动 |
